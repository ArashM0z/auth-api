03

Request lifecycle (Fastify hooks)

Every request runs this gauntlet before your handler — and the error surface catches anything that throws.

  1. onRequest → per-IP rate limit
    Redis INCR fixed window. Probes opt out via skipIpRateLimit. Over cap → 429.
  2. Content-type parser
    text/plain parser removed, so non-JSON → 415. Body capped at 16 KiB → 413.
  3. Schema validation (AJV)
    coerceTypes:false, additionalProperties:false — wrong types and unknown fields are rejected, not coerced/stripped. Fail → 400.
  4. Route handler
    Domain logic runs: normalize, policy, hash, Redis. May throw new ProblemError(...).
  5. onSend → X-Request-Id
    Correlation id echoed on every response (accepted if well-formed, else minted).
  6. onResponse → metrics
    Counter by method/route/status; route uses the template so cardinality stays bounded.
▸ Read it

Cross-cutting concerns are hooks, not per-route code, so they can't be forgotten on a route. The single error handler converts everything — validation, 404/405, 413/415, thrown problems, unexpected 500s — into one RFC 9457 shape.

▸ In one line

"Security-critical gates are framework hooks, so they apply uniformly. The AJV config is deliberately the opposite of Fastify's defaults — strict rejection, no coercion — because this is an auth API."

Mermaid source 
flowchart TD
  A[onRequest: per-IP rate limit] --> B[content-type check<br/>415 / 413]
  B --> C[AJV schema validate<br/>no coercion, no extra fields → 400]
  C --> D[route handler<br/>may throw ProblemError]
  D --> E[onSend: X-Request-Id]
  E --> F[onResponse: metrics by template route]
  D -.throws.-> G[setErrorHandler → RFC 9457 problem+json]
04

Create-user flow

POST /v1/users — normalize, policy-check, hash, then let Redis decide uniqueness atomically.

  1. normalizeUsername()
    NFC → trim → lowercase → charset. Invalid → 422 INVALID_USERNAME.
  2. validatePassword()
    NIST policy: ≥15 code points, blocklist, no username-containment. Returns all violations → 422 WEAK_PASSWORD.
  3. hasher.hash()
    Argon2id, through the concurrency gate. Hash happens before the write so a conflict costs the same as a success — no timing oracle.
  4. SET user:<name> … NX
    Atomic create-if-absent.

◆ reply not null

201 Created + Location: /v1/users/<name>

audit user.created

◆ reply null (key existed)

409 USERNAME_TAKEN

audit user.create_conflict

▸ Read it

The uniqueness decision is a single SET NX. A nil reply is the "already taken" verdict — there's no separate EXISTS check, so two racing signups can't both win.

▸ In one line

"I treated uniqueness as an atomicity problem. SET NX makes it a property of the datastore, not of application discipline — the classic check-then-set race is structurally impossible."

Mermaid source 
flowchart TD
  A[POST /v1/users] --> B{normalize username}
  B -->|invalid| E1[422 invalid-username]
  B -->|ok| C{password policy}
  C -->|violations| E2[422 weak-password<br/>all rules at once]
  C -->|ok| H[Argon2id hash<br/>before the write]
  H --> S[SET user:<name> json NX]
  S -->|reply ok| OK[201 Created + Location<br/>audit user.created]
  S -->|reply nil| CF[409 username-taken<br/>audit user.create_conflict]
05

Login — the timing-safe sequence ★

The centerpiece. Unknown-user, wrong-password, and bad-format all cost the same and return the same 401.

▸ Read it

Two things make this safe. The gate is an atomic INCR before the hash (a read-only peek would let a concurrent burst slip past the cap — that was the real TOCTOU bug the review caught). And the unknown-user path still runs a full Argon2id verify against a dummy hash, so timing can't tell the cases apart.

▸ In one line

"A wrong password and an unknown user are byte-identical in body, headers, and timing. The limiter increments before verifying, so it's both the brute-force cap and race-safe."

Mermaid source

06

Rate limiter & the TOCTOU fix

Two windows, both in Redis. One atomic MULTI per hit — and the ordering that closed a real race.

rl:ip:<ip>
All traffic, per source IP. 100 / 60s default. Enforced in the onRequest hook.
INCREXPIRE NXTTL
rl:login-failures:<user>
Login failures only, per submitted username (real or fake). 10 / 15min. Cleared on success.
INCREXPIRE NXTTLDEL

✗ the racy design (rejected)

peek → if count < max → verify → incr

A burst of guesses all read count < max before any increments land, so all slip past the cap. HIGH-severity TOCTOU.

✓ the shipped design

hit(): atomic INCR first, then compare to max

Redis serializes the increments, so at most max guesses ever reach the expensive verify. Window clears on success.

▸ Read it

INCR + EXPIRE NX + TTL run in one MULTI, so the window's expiry is set exactly once by whoever creates the key. State in Redis = the cap holds across every replica; an in-memory limiter would multiply the threshold by the instance count.

▸ In one line

"The adversarial review found a TOCTOU here: peek-then-check let a concurrent burst bypass the cap. The fix is to increment first and read the returned count — Redis serializes it. I added a concurrency regression test."

Mermaid source 
flowchart TD
  H["hit(policy, subject)"] --> M[MULTI: INCR key · EXPIRE key NX · TTL key]
  M --> Q{count <= max?}
  Q -->|yes| P[allowed → run Argon2id verify]
  Q -->|no| L[429 + RateLimit / Retry-After headers]
  P -->|login success| D["clear(): DEL key"]
07

Password hashing: the memory gate

Argon2id is memory-hard by design — so unbounded concurrency would be a self-inflicted DoS. A p-limit gate bounds it.

▸ Read it

Each in-flight hash reserves ~19 MiB. The gate caps that at 8 × 19 ≈ 152 MiB worst case; the rest queue. The hash_queued gauge is your scale-out signal — it rises while requests are merely slow, before they error.

▸ In one line

"Argon2's memory-hardness is the security feature and the operational risk. The p-limit gate makes worst-case hashing memory a bounded, predictable number instead of an OOM under load."

Mermaid source 
flowchart LR
  N[N concurrent hash/verify] --> G[p-limit gate<br/>maxConcurrency = 8]
  G --> A[active ≤ 8<br/>≈ 152 MiB worst case]
  G --> Q[queued<br/>authapi_hash_queued → scale out]
08

Redis data model

Three key shapes. That's the entire persistent footprint.

user:<normalized>
One JSON blob per user: { username, passwordHash, createdAt, passwordRehashedAt? }. Single key, not a hash + index — removes index-desync bugs.
SET NXGETSET XX
rl:ip:<ip>
Per-IP fixed-window counter. TTL = window. All traffic.
INCREXPIRE NXTTL
rl:login-failures:<user>
Per-username failure counter. Keyed on the submitted name so it can't probe existence. Cleared on success.
INCREXPIRE NXTTLDEL
▸ Read it

The stored record is username + hash + timestamps, nothing else — data minimization (PIPEDA). The whole user lives under one key so the SET NX uniqueness guarantee covers the entire record atomically.

▸ In one line

"Storing the user as a single JSON value under one key is what lets SET NX be the whole uniqueness mechanism. It also means there's no separate index to fall out of sync."

Mermaid source

Error model — one shape, every failure

Every non-2xx is application/problem+json (RFC 9457) with a stable code and requestId.

Body fails schema400VALIDATION_ERROR · errors[]
Not valid JSON400MALFORMED_BODY
Wrong / unknown login401INVALID_CREDENTIALS (identical)
Unknown path404NOT_FOUND
Wrong method on known path405METHOD_NOT_ALLOWED + Allow
Username exists409USERNAME_TAKEN
Body > 16 KiB413PAYLOAD_TOO_LARGE
Non-JSON content-type415UNSUPPORTED_MEDIA_TYPE
Bad username / weak password422INVALID_USERNAME · WEAK_PASSWORD
Rate limited429RATE_LIMITED + Retry-After
Unexpected500INTERNAL_ERROR (sanitized)
Overloaded / Redis down503SERVICE_UNAVAILABLE
▸ Read it

Registration failures list every violated rule at once (fix in one round trip). Login failure is the one place detail is withheld — a generic 401 — because uniformity there is an enumeration defense.

▸ In one line

"Error format is a standards-citation exercise, not an invention. One RFC 9457 shape everywhere, each with a stable machine-readable code and a requestId that correlates to the logs."

Component layers

A strict dependency direction — HTTP knowledge stays at the top, the domain knows nothing about Fastify.

server.tsboot, tracing, graceful shutdown (close-with-grace)
↓ builds
app.tswiring · hooks (IP limit, request-id, metrics) · RFC 9457 error surface
↓ registers
routes/users · auth · health · metrics — translate HTTP ↔ domain
↓ call
services/user-service — owns the Redis key layout: create, verifyCredentials
domain/username · password-policy · password-hasher · blocklist — zero HTTP knowledge
plugins/redis client · rate-limit — infrastructure adapters
▸ Read it

Nothing below app.ts imports Fastify reply types. Branded types (NormalizedUsername, PasswordHash) make crossing a layer with the wrong value — a raw username as a key, plaintext where a hash belongs — a compile error.

▸ In one line

"The domain is pure and testable in isolation; routes are the only place that knows about HTTP. Branded nominal types enforce the boundaries at compile time, not in review."

Mermaid source

Testing strategy

Five layers, plus mutation testing that checks the tests themselves. 79 tests, ~96% coverage.

Mutation testing — Stryker
mutates source, fails CI if tests don't kill it · 87% (gate 80%)
Contract
live responses vs committed openapi.json
Security attack-suite
each test is an attack that must fail
Integration — Testcontainers real redis:8
race, identical-401, rehash, both limiters, protocol edges
Unit + Property (fast-check)
policy · username · problems + adversarial Unicode fuzzing
▸ Read it

Wide base of fast unit + property tests; real-Redis integration for the things that only Redis semantics can prove (NX, MULTI, TTL). The cap on top is mutation testing — it measures whether assertions would actually catch a regression, not just which lines ran.

▸ In one line

"Coverage tells you a line executed; mutation testing tells you a test would fail if that line broke. A Stryker probe even deleted the dummy-verify — the timing test caught it, which proves the defense has a real regression guard."

Mermaid source
